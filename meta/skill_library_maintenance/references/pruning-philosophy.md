# 技能精简哲学 — 少即是多

## 核心原则

**不用 = 不存在。** 技能的价值在于被频繁使用，不是在于数量多。

## 三个层次

| 使用频率 | 处理方式 |
|---------|---------|
| **反复用** | 优化到极致。精简流程、补充陷阱、提炼快捷键 |
| **偶尔用** | 保持精简。去掉冗余描述，只留必要步骤 |
| **从没用过** | 直接删除。留着只会增加心智负担 |

## 用户原话

> "少就是多，宁缺毋滥。"
> "长期不用的技能就删掉，影响思维混乱的就删掉，保留主干。"
> "反复用到的技能要优化优化再优化到最佳模式。"
> "宁愿技能用不到、没有，也要保证经常用到的技能优秀、优质、有效、高效。"

## 2026-07-15 实操记录

清理前：271 个技能
清理后：35 个技能（-87%）

**删掉的类别：**
- gstack 生态全家桶（~40 个，从没用过）
- 创意设计类（~20 个，ascii-art/manim/p5js 等）
- ML/AI 类（~15 个，llama-cpp/vLLM/DSPy 等，不在本机跑）
- 游戏/社交/社区/邮件/智能家居等一次性技能

**保留的核心：** 问AI / 日记 / 状态管理 / 微信管道 / Obsidian / 舰队监控 / DevOps 排错

## 后续维护原则

- 新技能加入前先问：**这个真的会被反复用吗？**
- 如果创建后一个月没用过 → 自动标记待删
- 定期审计（每月）：查使用率，删从未触发的
