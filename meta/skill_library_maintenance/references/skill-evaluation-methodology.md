# Skill 触发率评估方法

## 原理

Skill description 写得越准，AI 触发越精准。评估目的：量化 description 的召回率（应该触发的都触发）和精确率（不该触发的不误触）。

## 4 步闭环

```
写/改 Skill → 跑 Eval → 看数据 → 改进 → 再跑 Eval
```

### Step 1: 准备测试集

各 10 条，共 20 条：
- **10 条应该触发的**（覆盖各种可能的自然语言变体）
- **10 条不该触发的**（邻域干扰项，如 ping、翻译、文件操作等无关任务）

示例（以 diffusion-convergence skill 为例）：

应该触发：
```
"我想做个MCP，把日志服务统一管理起来"
"这个方案行不行？帮我分析一下"
"有个问题我想不通，帮我理一理"
"帮我规划一下自动化部署体系"
"我想建一个联邦架构，但不知道从哪开始"
```

不该触发：
```
"帮我ping一下192.168.1.8"
"今天天气怎么样"
"帮我查一下ip地址"
"写一个hello world程序"
"翻译这段英文"
```

### Step 2: 运行评估

使用关键词匹配模拟（置信度较低的测试——真实 Hermes 用语义匹配更准）：

```python
keywords = ["模糊", "方案", "规划", "分析", "设计", "不知道", ...]

def fires(query):
    q = query.lower()
    return sum(1 for kw in keywords if kw in q) >= 2
```

### Step 3: 计算指标

| 指标 | 含义 |
|------|------|
| TP | 该触发且触发了 |
| TN | 不该触发且未触发 |
| FP | 不该触发但触发了（扰动） |
| FN | 该触发但未触发（漏召） |
| 准确率 | (TP+TN)/总数 |

### Step 4: 修补 description

漏召（FN）→ description 缺相关关键词，补上
扰动（FP）→ description 泛化过宽，收紧

**修补后重新跑一轮，确认改进。**

## 输出质量测试（进阶）

触发率达标后再测输出质量：

1. 用 3-5 个真实场景执行 Skill
2. 对比「带 Skill」vs「不带 Skill」的输出差异
3. 检查：规则是否被遵循？输出结构是否符合预期？
4. 根据缺陷迭代 SKILL.md 正文
