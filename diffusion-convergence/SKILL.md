# 扩散收敛法 (Diffusion-Convergence Method)

## 触发条件

当用户有一个**模糊想法/问题/目标**，需要从「说不清楚」到「可执行方案」时。

典型场景：
- 「我想做个 X，但不知道从哪开始」
- 「这个方案行不行？帮我分析一下」
- 「有个问题我想不通」
- 「帮我规划一下 Y」

## ⚠️ 硬性前置：先调查再动手

处理任何技术问题前，必须按此顺序操作。不要直接动手、不要扩散、不要试错：

| 步骤 | 做什么 | 原因 |
|:----:|--------|------|
| 0️⃣ | **读说明书** — 该工具的官方文档/README/Wiki | 绝大多数问题说明书能解决 |
| 1️⃣ | **搜已知问题** — GitHub Issues、论坛、社区讨论 | 别人踩过的坑不用再踩 |
| 2️⃣ | **问 AI** — 整理好上下文问 ChatGPT/Claude/Gemini | 上面两步都解决不了再问 |
| 3️⃣ | **动手** — 前三步完成后才操作 | 减少试错、节省 token |

> 用户铁律：「你每次配置啥都失败，就是事前调查没做好」「磨刀不误砍柴工」

**如果上一个步骤就解决了，后面的步骤不需要执行。**

## 核心原则

**先扩散，后收敛。** 一散一收为一个循环，可多轮循环逐步细化。

### 前置原则：先调查再动手

在处理任何技术问题时，**不要直接开始扩散或动手操作**。先做调查：

1. **读说明书** — 该软件的官方文档/README/Wiki 先通读一遍
2. **搜已知问题** — GitHub Issues、论坛、社区讨论搜索同类故障
3. **问 AI** — 上述两步后仍卡住，再问 ChatGPT/Claude/Gemini（整理好完整上下文）
4. **自己试** — 以上三步完成后再动手

> 用户铁律：「绝大多数问题你看说明书就能解决」「你每次配置啥都失败，就是事前调查没做好」

| 阶段 | 目标 | 动作 |
|------|------|------|
| 扩散 | 铺开认知地图，暴露盲区 | 列出所有相关维度、分叉、方案选项、隐含依赖 |
| 收敛 | 锁定确定性，缩小范围 | 每次只问一个关键问题，决定后锁定该决策，继续下一轮 |

**节奏控制**：
- 每一轮 AI 只问**一个问题**（最关键的分叉）
- 每个决策点用表格对比两三个方案
- 推荐方案标置信度（如「推荐 B 方案，置信度 85%」）
- 用户回答后锁定决策，进入下一轮

## 工作流

### 第一轮：扩散初始化

拿到用户的模糊输入后，先做一次扩散：

1. **拆解维度** — 列出问题涉及的所有维度（技术栈/架构/成本/风险/人力等）
2. **暴露暗礁** — 标出可能影响设计的隐性约束
3. **方案地图** — 列出主要的路径分叉
4. **置信度说明** — 对自己有把握的和猜测的分开标注

### 收敛循环

每轮只问一个关键分叉。格式：

```markdown
## 第 N 轮收敛：{决策主题}

### 选项对比
| 维度 | 选项 A | 选项 B | 选项 C |
|------|--------|--------|--------|
| {标准1} | ... | ... | ... |
| {标准2} | ... | ... | ... |

**推荐方案：选项 B**（置信度 85%）
理由：{简明解释}

**我的问题：**
{一个问题，唯一的关键分叉}
```

用户回答后：
1. 记录决策到**决策日志**
2. 锁定该方向（不再回头讨论）
3. 进入下一轮收敛（下一个未锁定分叉）

### 决策日志

每轮锁定的决策追加到日志末尾：

```markdown
## 决策日志
| 轮次 | 主题 | 决策 | 理由 |
|------|------|------|------|
| 1 | 技术栈 | Python + FastAPI | 团队最熟，MVP 最快 |
| 2 | 存储 | SQLite → PostgreSQL | 数据量大后迁移 |
```

### 输出物

全部分叉收敛完毕后，输出一份 **MVP 方案文档**，使用以下结构：

```markdown
# {项目} MVP 方案

## Problem Statement
{要解决的问题，从用户视角描述}

## Solution
{解决方案一句话概括}

## User Stories
1. 作为<角色>，我想要<功能>，以便<收益>
2. ...

## Architecture Decisions
- 决策1：{选择了什么} — 理由
- 决策2：{选择了什么} — 理由
（来自决策日志）

## Implementation Plan
{可被任意 AI/人接手的执行步骤}

## Testing Strategy
{如何验证方案正确}

## Out of Scope
{明确不做的内容及理由}
```

## 关键约束

1. **一次一个问题** — AI 不得在单轮输出多个问题让用户选，那等于把思考工作推回给用户
2. **推荐优先** — 每个分叉必须有推荐方案，不能只列选项让用户自己挑
3. **锁定不回头** — 锁定后的决策不再重新讨论，除非新证据出现
4. **模糊接受** — 如果用户对某维度还没想清楚，先记「待定」并继续，不卡死
5. **置信度标注** — 每个推荐必须标置信度，让用户知道 AI 的把握程度

## 示例流程

```
用户: 「我想做个 MCP，把我的日志服务统一管理起来」

→ AI 扩散: 拆出 6 个维度（协议/部署/安全/存储/成本/运维）
→ AI 问: 「MCP 用 stdio 还是 HTTP？HTTP 可以远程调用但部署更复杂」
  用户: 「HTTP」
→ AI 锁: 协议=HTTP
→ AI 问: 「服务端用 Python 还是 Go？Go 性能好但 Python 原型快」
  用户: 「Python」
→ AI 锁: 语言=Python
... 继续直到全部分叉锁定
→ AI 输出: MVP 完整方案（架构图+目录结构+关键代码+部署步骤）
```
