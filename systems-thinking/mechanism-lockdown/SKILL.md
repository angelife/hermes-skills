# Mechanism Lockdown（机制锁死）

用户的核心方法论原则。当发现规则执行失败时，先问"是因为规则不够是因为入口藏得太深"——如果是后者，不要加新规则，把旧规则搬到必经路径上。

## 核心原则

**机制锁死 > 行为规范。**
**必然 > 概率。**
**必经路径 > 额外步骤。**

| 行为规范（避免） | 机制锁死（追求） |
|-----------------|-----------------|
| "记住要在每次交互后检查..." | 写一个 cron job 自动检查 |
| "这条规则存在 memory 里，我每次会读" | 规则写在系统提示注入层、SOUL.md 或外部代码路径 |
| "我确保下次不会先猜测用户意图再行动" | 写进 response protocol 的 pitfall 里，加上反论证开头强制自检 |
| "flag 通知我更新配置" | 'hermes config set' 自动路由 |

## 诊断框架

当任何规则/流程/步骤重复失效时，问有序的三个问题：

1. **这是规则问题还是入口问题？**
   - 规则不清晰/不完整 → 改规则
   - 规则本身是对的但藏在需要主动想起的地方 → 搬到必经路径
2. **必经路径是什么？**
   - startup 注入（SOUL.md / system prompt）
   - 工具注册层（每个 import 自动注册）
   - cron job（定时触发，不依赖记得做）
   - conversation_loop post-processing（每次交互后自动执行）
3. **行为规范的补充角色是什么？**
   - 用于**过渡期**（新机制未建好前的临时措施）
   - 用于**观察期**（先输出建议不执行，验证机制准确后再切自动）
   - 永远不是**最终方案**

## 三阶段 rollout 模式

适用于任何自动化决策（记忆管理、cron job、规则执行）：

```
阶段 1: 观察模式（cron 写日志，只建议不执行）
阶段 2: 抽检验证（用户检查日志，确认规则判断准确）
阶段 3: 自动执行（规则经过验证后，切到自动）
```

**关键约束：** 删改是不可逆操作（memory 被删后无法恢复）。所以阶段 1 是必须的——规则第一版大概率有漏判，先自动执行会把误删代价转嫁到用户身上，而且用户不会知道删错了（因为被删了）。

## 案例库

### 案例 1：Memory 审计（2026-07-03）

- **问题：** Memory 到了 83% 才被意识到需要压缩。没有自动机制。
- **错误方案（行为规范）：**
  - (a) "每次交互后手动检查记忆" ← 无即时收益，被系统性跳过
  - (b) "把审计规则存在 memory 里" ← 规则本身也会过期/被删
- **正确方案（机制锁死）：**
  - cron job 每天 03:00 跑审计，写日志
  - 先跑观察模式（写日志，不修改 memory）— 默认 7 天
  - **自动切换：** 第 7 天自动切到执行模式，不依赖用户记得检查日志（定时+状态文件自检）
  - **安全阀：** 如果 cron job 连续 7 天标记了同一件判断不了的问题，设 pause_on_error 暂停自切，通知用户介入
  - **日志自清：** cron job 每次跑先删 30 天前的旧日志，不让日志本身变成第二个膨胀点
  - 保鲜触发器（每天一次）和紧急触发器（>80%容量）分离

**关键纠正（2026-07-03）：**
  - ❌ 把"抽检确认后再切"作为方案——这是行为规范，不是机制锁死（依赖用户记得去看日志）
  - ✅ 切换条件写死在 cron job 里：定时 7 天，有安全阀兜底，不等人批准
  - ❌ 只关注审计目标（memory）而忽略审计产物（日志）——机制产生的副产物也必须自带过期规则
  - ✅ 日志保留期写进 cron job 第一步：`find -mtime +30 -delete` 先于写日志

### 案例 2：SOUL.md 身份锁定

- **问题：** 跨会话身份漂移（被用户提示或对话历史改写角色）
- **方案：** 身份不放在 memory 里（会被覆盖/删），放在 SOUL.md 中通过 startup 注入。每次会话加载，不可被 user message 改写。
- **原则应用：** 身份是"必经路径"（startup chain），不是"记得自己是谁"（memory）。

### 案例 3：先查文档再动手（2026-07-16）

- **问题：** Kodi 崩溃——没读说明书就暴力卸载，不知道 Kodi 21 Omega 的 IPTV Simple 有已知 timeshift 死锁（GitHub #760），读文档后发现全是已知问题。
- **行为规范的无效性：** 告诉自己"下次先读文档"——下次还是不会读。
- **机制方案：**
  - 工具清单 `tool-inventory.md` 标出理解度%，低于 60% 的工具禁止直接操作
  - 先问AI（SOUL.md 已写入）强制触发：操作前问 AI → AI 要求读文档 → 读完再动手
  - 每晚读一个工具说明书，理解度写在 Obsidian 工作档案，不依赖 memory

### 案例 4：问题解决层级（2026-07-16）

- **问题：** 遇到问题就自己硬搞，浪费 token 和时间。
- **层级锁死：**
  1. 读该工具的官方文档（第一手权威）
  2. 查 GitHub Issues / 社区讨论（别人踩过的坑）
  3. 再搞不定 → 问 ChatGPT / Claude / Gemini

## 判断检查

设计任何方案后，跑这道检查：

1. 如果我**忘记**执行它，它会自己执行吗？ → 是机制，否行为
2. 如果我这一轮 session memory 被清空，这条规则还在吗？ → 是机制，否行为
3. 如果我换了模型/provider/终端，这条规则还在吗？ → 是机制（外部代码），否行为（内化规则）

三项全"是" → 机制锁死成功。
任意项"否" → 至少部分回到了行为规范，需进一步加固。
