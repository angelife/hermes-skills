# minimum-first-protocol — 先解决唯一触发的、别预先做盘点

**适用情况：** 收到一个 task 包，但 task 包的范围大于"目前紧急的唯一触发点"。

## 起源（已发生的典型失败模式）

一例实证：本轮，用户实际问"金同学当前 deepseek-v4-flash 是不是真免费"，**唯一触发**就是这一个 model。我**默认**做 65 个 model 跟 memory 名单比对——这是"曾记录过" vs "现在仍然有效" 的两种混用，**典型的预先盘点**。被立即打断，重定方向。

## 强制规则

### 不要做 65 个的盘点（除非那是用户的真实 spec）

- 用户问 X → 只查 X 是否 X
- X 还没验真之前不去主动扩到 Y, Z, ...
- 后续要不要扩，**等 X 验真后再判断**

### "memory 名录" ≠ "现状"

三种误判：
1. memory 记过免费，但 provider 后来改了 → 误判成免费
2. memory 没记，但其实一直免费 → 误判成不免费（漏掉可用资源）
3. 名字相近但版本不同 → 不能靠"名字像"当同一件

→ memory 只用来**启发**搜索关键词，**不**作结论依据。

### 范围不止反例——一切 task 都有"最小在场"

- 收不到 task 包的全貌时，给出**最小在场**的解：先验当前最迫切的具体子问题，等该子问题有结果再判断扩展边界。
- "在唯一触发的 model 上做完整可能性展开" 是错解。
- 在不掌握完整 spec 之前，"做 65 个"是对范围假定的乐观估计。

## 反例（错误做法）

1. 用户："金同学 /model 哪些真免费？"
   → 助理读 /v1/models 后开始做"65 个 vs memory 名单"大盘点
   → 实际用户只是要先确认**当前在用的**那一个是否真免费

2. 用户："Cron 还报错么？" 
   → 助理做大范围 "cron fail/pass aggregate" ——其实用户是在问**上周指定的某一个具体** cron

3. 用户："金同学说话不正常么？"
   → 助理列 5 个可能 diagnosis 各自开走 —— 实际用户是要"够用"的当前判断

## 正确做法（示例）

用户问 X → 立即精确化 X 的最小定义 → 解 X → 给"要不要扩"一句话末尾，不预先做。

```
Q: 金同学 /model 里哪些真免费？

A: 范围先限定在最低问题：金同学当前 model.default = deepseek-v4-flash，
   只查这一个。后续是否扩到其他 model，待"deepseek 是否免费"验真有结果
   之后单独决定。
```
