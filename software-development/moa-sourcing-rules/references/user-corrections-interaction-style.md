# 用户交互偏好：测试与验证先行

## "测试X的来回" = 先测试，不分析

**触发场景：** 用户说"测试X的来回"、"test X"、"试一下X"。

**这不是诊断请求，是执行请求。** 用户要的是你动手测然后报告结果，不是你分析"测之前这个可能哪里有问题"。

**正确做法（一次过）：**
1. 立即执行测试（发送消息 / 跑脚本 / curl endpoint）
2. 一行报告结果（"消息已送达 message_id=X" 或 "失败: Y"）
3. 只有测试失败才进入诊断模式

**错误做法（本会话验证，重复5次）：**
```
User: 测试土同学的来回
Agent: gateway状态分析、日志解读、proxy检查、长文报告...
User: 测试土同学的来回
Agent: 更多的分析...（发送了一次但被分析文字淹没）
User: 测试土同学的来回
```

用户重复同一指令说明响应格式错了——不是在补充上下文，是在告诉你"别分析了，测"。每轮分析都比直接执行测试贵。

**关键区别：**
- 测试 = 执行操作 + 简洁报告结果（便宜路径，先做）
- 诊断 = 调查哪里坏了（贵路径，测试失败才做）

这与"测试尽可能在自己侧执行"偏好的关系：测试先行是"做什么"，本地执行是"在哪做"，两者搭配不冲突。

## "你本地自己就可以测试" — 测试尽可能在自己侧执行

**触发场景：** 当需要测试 API 连通性、模型可用性、配置正确性时，如果本地环境（Mac）有相同的凭证或可达性条件，**先在自己环境测试**，而不是让用户在远程环境上执行命令贴结果。

**这个偏好的触发条件：**
- API key 在本地 config 中有记录 → 本地先 curl 测试
- 远程环境不通（如 WebSocket）→ 本地可以分析问题而不需要用户重复执行
- 需要验证模型/API 是否工作 → 本地先确认，再到远程配置

**例外（必须请用户执行）：**
- 远程环境独有变量（如 `PROXY_URL`、`HW_ACCESS_KEY`）
- 远程环境的网络拓扑不同（内网可达 vs 外网）
- 需要用户登录/注册的操作

## "不是 我怎么感觉这个不是免费的呢" — 验证"永久免费"声明的可信度

**触发场景：** 推荐免费资源（API、模型、服务）给用户前。

**必须做：**
- 区分"新人试用额度"（如阿里云 DashScope 100万/月）和"永久免费"（如智谱 GLM-4-Flash）
- 如果声称永久免费，必须确认该声明的来源（官方文档、个人实测）
- 如果有免费活动截止日期，必须标注

**禁止做：**
- 把"有免费试用期"包装成"免费"推荐
- 不确认额度是否再生（月刷新 vs 一次性）

## "不是你还是没搞明白" — 架构分层理解不到位时要停

**触发场景：** 涉及多层架构（基础设施层 vs 用户层）的讨论。

**必须做：**
- 区分：这是平台基础设施的能力 vs 用户 session 的限制
- 如果被纠正过类似错误，先查 memory/skill 确认理解正确再继续说
- 不确定时，直接问用户"你是说 X 还是 Y？"而不是强推自己的理解
