# crisis-validation-protocol — fix 后真伪验证的标准化流程

**适用情况：** 修完 bug / 改完 config / 重启完服务后，下一步是要"已修"上报还是"未修"再查的判断时刻。

## 起源（已发生的典型失败模式）

1. curl 直打上游 API 拿到 200 → 错误地标为 "已闭环"
2. transcript 里某段写着"X cron 在跑时报 Y 错误"但没核证到任何 cron log
3. 单次自然触发看到"不再是那个错"就当作"原描述不成立"
4. 看到 control panel 不报错（绿点）就当作"已修复"

**共同根源：** 单一证据撑一个强结论。

## 强制规则

### 1. 接口通 ≠ 业务流程通（必须分两条报）

**禁止做法：** 单证据就等同 "已修"。
- `curl 200` 仅证明上游可达。
- 进程在跑 ≠ 对用户可见功能工作。
- log 没新错误 ≠ 不出问题（可能没在打 log）。
- health check 是绿点 ≠ 实际数据流回得来。

**正确做法 — 分两条以不同标注报告：**

| 维度 | 含义 | 验证手段 |
|------|------|----------|
| ✅ 接口 / 技术路径 | URL / 模型 / provider 本身能否调通 | curl 直打、stack-trace 静态分析、单元测试 |
| ❓ 业务流程 / 用户可见 | 这条路径上**实际触发**后能不能跑通 | 自然触发 cron / 端到端任务 / 真实流量回测 |

**❓ 不允许被 ✅ 替代；必须显式说"业务流尚未验"。**

### 2. 验证标准必须**提前写到纸上**

**正确做法：** 任何"以下哪个结果算 X"问题 → 在去看观察结果之前先把判定逻辑写下来。写到 state 目录或工作备忘里。

**反例作为锚点：** 本轮"7003 是不是 cron 真报的"——验证标准只在我脑子里。下次去看 cron 日志时**会临场创造标准**，重犯同样的错。补救方法：**写 memo 文件先**——

memo 模板（写到 `~/.hermes/state/<task>-validation.md`）：

```markdown
# <任务> 验证标准（YYYY-MM-DD HH:MM）

## 目的
<具体要验的是什么，限定到边界>

## 抽样规则
- 样本数：≥N（一般 2）
- 抽样方式：自然触发 X 次，间隔 Y 分钟
- 拒绝手动 fire（破坏产线观察）

## 三档判定

| 两次观察结果 | 结论 |
|---|---|
| 都报 X | transcript 描述成立，接下来 ... |
| 两次报别的一致 | transcript 描述不成立；X 病本身要另查 |
| 不一致或其他 | 未定，继续下一次再下结论 |

## 反例分支（防再犯）

- "下一次不再 X" 不等价于 "X 描述不成立"
- 也可能 cron 不稳定撞上另一个独立 bug
```

### 3. 至少 ≥2 独立样本才下结论

- 不允许用单次观察下结论
- 手动 fire 不算"自然触发"；破坏观察
- "下一次不再是这个错" ≠ "原描述错"

### 4. transcript ≠ 事实

**触发情境：**会上轮拿下来的 transcript 片段，本轮开头当成"已发生过的事"继续建。

**正确做法：**
- 任何"上次 X 了"的引用 → 必须先 grep / 查文件 / 查 memory 验证
- 与"自我引用核实"规则联动使用
- 这条同样适用于：本轮观察到一项症状，断定其根因历史上是某事——但没核验过这个历史链。

### 5. WRITE INTEGRITY PROTOCOL（修改型动作的三步走）

1. **改前点头**——明说"准备以 X 方案改 Y"，等用户"确认" 才动
2. **改后核验**——不只看到工具返回"已修改"就当真，重新 `grep` / `cat` / `diff` 看到原文是目标值
3. **改后真实流回测**——改之后这个路径上**真下一次请求**，验证接口通

**sub-rule：编辑器工具（sed/awk/patch）后必须完整 diff 全检，不是抽查**

- BSD sed（macOS 默认）在 `c` / `a` / `i` 等命令上的语义与 GNU sed 不同——同条命令拷贝可能吞掉相邻行
- 任何 sed 编写后必须 `diff -u <backup> <current>`，且用 `python3 -c "import yaml; yaml.safe_load(open(path))"` 重新解析 YAML —— YAML 吞键而**看起来没坏**
- 跨同一配置文件有同名条目（如同一 provider_name 既在 `providers:` 又在 `custom_providers:`），**必须两处都改**

## 没踩过的、但提醒要经常用

- `process running ≠ bot working` —— 老生常谈
- macOS / Android / Linux 多平台下，"绿点"意义的差异：macOS launchd 的"绿"等价于"进程启动了"，不确认是否完成初始化；Android 的 `STATE_RUNNING` 等价。
- application-level 的 200 健康检查可能因为上游只返 200 但内容错乱（如 FreeLLM-API 那种返 HTML 的 bug）。
