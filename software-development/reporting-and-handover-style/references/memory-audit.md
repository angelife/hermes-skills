# Memory Audit Cron Job Pattern（2026-07-03）

## 背景

Memory 管理从"行为规范（记得压）"改为"机制锁死（cron job 每天自动检查）"。

## 配置详情

- Hermes cron job ID: `226e5ad24d64`
- 名称: `memory-audit-observation`
- 周期: 每天 03:00 UTC+8
- 模式: 观察模式（只写日志，不执行删除）
- 日志路径: `~/.hermes/cron/output/memory-audit-YYYYMMDD.txt`

## 工作方式

Cron job 以 AIAgent 身份运行，从注入的 MEMORY 段中读取当前条目，按以下规则分类：

1. 读取当前容量和条目数
2. 对每条分类：[PREF] 用户偏好 / [ENV] 环境事实 / [TECH] 技术方案 / [TODO] 待办 / [PROC] 流程
3. 按规则生成建议：TODO 类标记删除、TECH 类含参数路径标记缩短、容量 >70% 标记压缩
4. 写审计日志到文件

## 三阶段 rollout

| 阶段 | 模式 | 操作 | 时间 |
|------|------|------|------|
| 1 | 观察（当前） | 只写日志，不删不改 | 1-2 周 |
| 2 | 验证 | 用户抽查日志，确认分类准确 | 持续 1-2 周 |
| 3 | 自动执行 | cron 获得 memory 写权限，按规则 add/replace/remove | 阶段 2 通过后 |

## 规则（等待验证的边界情况）

1. **7天未读删除**：什么算"读取"？跨设备读取算不算？被动读取（提到话题触发）算不算？这些在阶段 1 需要从日志中发现并定义。
2. **矛盾修正**：如何自动检测与新条目矛盾的旧条目？目前依赖 cron agent 的分析能力，无程序化 diff。
3. **80% 触发 vs 每天保鲜**：两个独立触发器。80% 是 reactive（容量压力），03:00 是 proactive（定期保鲜）。不应合并。

## 从这轮会话沉淀的压缩标准

| 规则 | 说明 | 示例 |
|------|------|------|
| 7天未读且未被引用 | 条目在 Hindsight 中也无法匹配到最近话题 | 节点状态（运行稳定后） |
| 版本号/状态记录 | "Hermes v0.18.0" 类信息，升级后即过时 | 升级前版本号 |
| 一次性诊断参数 | 具体 IP/端口/命令路径 | RNDIS IP 具体值 |
| TODO/打算做的事 | "计划要做 X" 类条目 | "Token 用量日统计" |
| 高价值保留 | 用户偏好、协议、方法论、关键警告 | 回应协议、adb install 警告 |
