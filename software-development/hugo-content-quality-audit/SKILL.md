# Hugo 内容质量审计

## 目标

批量识别并剔除 Hugo 站点 `.md` 文件中的 AI 生成模板文本，恢复作者原意与语言密度。

## 违规模式库

### 高频模板（必扫）

| 模式 | 位置 | 示例 |
|---|---|---|
| `## 结语` + 模板段 | 文章尾部 | `## 结语\n\n本文从5个角度探讨了...` |
| `## 题记`（空/占位） | 文章头部 | `## 题记\n\n在信息与经验之间...` |
| `涉及多个层面。关键是行动与反思的结合` | 文章尾部 | 固定句式，无实际信息 |
| `在信息与经验之间，真正的瓶颈不是数据本身` | 题记下方 | 占位语，重复出现在多篇 |

### 中频模板

- `本文从X个角度探讨了「\"...\"」。信息过载时代，真正的能力不在于掌握更多数据，而在于判断的准确性。`
- 重复段落（同一句出现两次以上）
- 重复链接（同一参考链接列出多次）
- 重复署名行

## 执行流程

1. **扫描定位** — 用 `search_files` 逐一匹配违规模式，记录命中文件列表
2. **逐文件读取** — 对每个命中文件，用 `read_file` 读取含上下文的尾部/头部，确认完整模板块
3. **精准 Patch** — 用 `patch` 删除整个模板块（含标题+正文+空行），不修改真实内容
4. **二次扫描** — 重新搜索所有模式，确认清零
5. **Hugo 构建验证** — `hugo --quiet` 确认零错误

## Patch 技巧

- **空 `## 结语`**：删除 `## 结语\n\n` 及其后的模板段落，保留前文最后一句真实内容
- **空 `## 题记`**：直接删除 `## 题记\n\n` 及占位内容，如果下方有 `## 零、本文结构` 则一并删除空题记保留结构头
- **尾部模板**：注意模板前可能有空行，patch 时连同空行一起删
- **replace_all=False**：除非确认全文只有一处，否则不用 replace_all——避免误伤
- **连续 patch 失败**：如果 `old_string` 匹配不到，先用 `read_file` 读取尾部确认精确文本再重试

## 坑

- **`replace_all=True` 是陷阱**：某些文件（如旧 Blogger 迁移文）中模板字符串可能出现在正文内多处，全局替换会破坏正文。**永远用 replace_all=False，靠上下文让 old_string 唯一**
  - 案例：zhiyuan-fuwu-lifa-yanjiu.md 使用 replace_all=True 后整个文件内容被替换成模板字符串拼贴，必须 `git checkout` 恢复
- **Python subprocess 被 block**：`execute_code` 不允许运行 `subprocess`。如需批量处理请用 `terminal` 调用 Python
- **空响应拦截**：连续 patch 后必须附带实质性文本输出，否则系统会反复提示空响应
- **多行匹配**：模板通常跨多行（标题+空行+正文+空行），`old_string` 必须包含完整块
- **文件被外部修改后 stale**：如果 patch 返回 "Found N matches"，说明上下文不够唯一；如果返回 "unchanged"，说明文件已被其他操作修改过，需重新 read_file
- **`## 结语` 下无内容**：有些文件 `## 结语` 后只有空行没有模板段，这种也需删除——空标题本身就是违规
- **同一文件有多个模板实例**：某些旧文（如 24-voices-on-cult-management.md）在正文中间有一个 `本文从4个角度探讨了…`，在尾部又有一个 `## 结语`+`本文从6个角度探讨了…`。需要全部移除
- **恢复被 patch 损坏的文件**：`git checkout -- <file>` 快速恢复，然后重新用更精确的 old_string 重试
- **GitHub 推送被拦截**：如果 `git push` 被 GH013 拒绝，原因是提交中包含了 API key。用 `git log --oneline -- <file>` 定位对应提交，修改或删除该文件后 `git commit --amend` 重推
- **在正文中间找到模板**：不是所有模板都在文件尾部。有时模板突兀地出现在段落之间，要看上下文判断是不是占位语还是作者真要。案例：24-voices-on-cult-management.md 在正文中间有一个 `本文从4个角度探讨了…`，在尾部又有一个 `## 结语`+`本文从6个角度探讨了…`。需要全部移除。
- **GitHub 推送被拦截**：如果 `git push` 被 GH013 拒绝，原因是提交中包含了 API key。用 `git log --oneline -- <file>` 定位对应提交，修改或删除该文件后 `git commit --amend` 重推。案例：test_mi6_connectivity.py 中的 OpenAI key 导致推送失败。
- **同一文件多个不同版本模板**：某些旧文在同一文件中出现多次模板实例，每次 old_string 必须唯一，不能用 replace_all。
