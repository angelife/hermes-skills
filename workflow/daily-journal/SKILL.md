# Daily Journal — 土同学每日日志

## 触发条件

- 🌅 **早课**：每次新会话第一件事（手动）
- 🌙 **晚课**：每天 21:00（cron job `3f931158a125`）
- 💬 用户说"写日志"/"写日记"/"工作记录"
- 🔁 **犯错反刍**：犯错时即时触发

## 目标目录

**`~/Documents/Obsidian Vault/每日工作记录/YYYY-MM-DD.md`** — 如果 vault 存在

## 🌅 早课流程（每次新会话第一件事）

1. **通读 SOUL.md** — 逐条理解，问三个问题：
   - **为什么定这条？**（防止过什么错误）
   - **怎么应用？**（今天什么场景可能触发它）
   - **哪里没做到过？**（犯过哪些相悖的错误）
2. **写下今日目标分数**（满分 100）

## 🌙 晚课流程（21:00 cron）

0. **前置检查：** 确认 `~/Documents/Obsidian Vault/` 是否存在。不存在则 `deliver='origin'`，所有输出直接推给用户；存在则写到 vault 文件
1. session_search 今天所有对话回顾活动
2. 翻犯过的错误 → 回 SOUL.md 找违反哪条
3. **评分：** 符合规则 +10 / 违背规则 -10 / 每日记录分数变化
4. **编译问题清单** — 整理当天所有未解决的问题，按优先级排序
   - 🔴 阻塞中（需要用户决策）
   - 🟡 诊断中（需要进一步验证）
   - 🔵 待归档（已有方案）
5. **记录三通道使用情况** — 今天用了哪条信息收集通道：
   - A: OpenBridge 问 AI（ChatGPT/Gemini/Claude）
   - B: 搜索→NLM（web_search → web_extract → NotebookLM）
   - C: Grok API / MoA
6. **昼夜分工** — 推不需要用户决策的事，整理需要用户拍板的待办
7. **输出：** 如果 vault 不存在，直接推送报告内容（`deliver='origin'`）；如果 vault 存在，追加到 Obsidian 每日工作记录末尾

## 📁 每完成一项任务：归档到工作档案

完成任何非琐碎任务（修好一个工具、配置一个系统、学会一个新功能）后，写一页笔记到：

**`~/Documents/Obsidian Vault/土同学工作档案/<工具名或任务名>.md`**

笔记结构：
```markdown
---
title: "任务/工具名 — 日期"
tags: [相关标签]
状态: 完成/进行中
---

# 标题

## 背景
为什么要做这个

## 做了什么
步骤、操作、关键命令

## 踩了什么坑
遇到的错误 + 解决方法

## 理解要点
这个工具的核心概念、架构、注意事项

## 关联知识
相关的其他工具/概念
```

> 用户要求：「所有做过的事情都要文字化。将来翻阅一看就明白还做过这些事。」

## 📗 晚修（晚上无事时的自学活动）

当用户离线且没有待办任务时，优先做：

1. **读一份软件说明书** — 从工具清单中选择未吃透的工具，通读其官方文档
2. **读 GitHub 仓库** — 开源工具的 README、Issues、Wiki
3. **写分析笔记** — 新学的内容整理到 Obsidian（土同学工作档案/）
4. **技能优化** — 检查现有技能是否过时、是否有新陷阱可补充

优先级：最常用的工具优先（Hermes > Kodi > Obsidian > v2rayN > nlm）

## 🧠 文章分析管线（来源：微信公众号/网页）

当有新的文章需要精读时（已归档到 Obsidian），按以下流程处理：

```
全文备份（已有） → 精读全文 → 评级 ★1-5 → 适用性评估 → 分析笔记 → 交叉链接
```

每篇文章产出：
- **评级**：内容质量 ★★★★★
- **适用性**：对我们系统的实际参考价值 ★★★★★
- **分析笔记**：核心论点、关键概念、与我们系统的关联、可落地改进
- **交叉链接**：引用 Obsidian 内其他相关笔记 `→ [[]]`

分析笔记文件命名：`<原标题>_分析笔记.md`，放在原文旁边。

## 🔧 工具掌握度追踪

工具清单位于 `~/.hermes/tool-inventory.md`，记录每个工具：
- **用途**：用来干什么
- **理解度**：0-100%（基于官方文档通读程度）
- **优先级**：P0（核心）P1（常用）P2（不常用）
- **计划**：哪个晚上啃哪个说明书

每天至少理解一个工具，理解度（%）记录在 Obsidian 工作档案。

## 🔁 犯错反刍（即时）

错误 → 翻 SOUL.md → 找违反的规则 → 写日志 + 扣分

## 🔍 跨 Session 监督评价（对其他 Session 的 SOUL.md 合规评估）

当被要求监督/评价其他 Session（同 profile 的其他对话线程，或另一台设备上的 bot）时，使用 八耻八荣 框架打分。

### 评价标准：八耻八荣（八条，每条约占 12.5 分，满分 100）

| 耻 | 荣 | 含义 |
|------|------|------|
| 瞎猜事实 | 查证溯源 | 不知道就查工具/文档/code，不靠记忆推测 |
| 模糊执行 | 澄清边界 | 需求模糊、条件冲突时，先问清再动手 |
| 擅自臆断 | 人类拍板 | 不可逆操作必须由人决策 |
| 另起炉灶 | 沿用现成 | 已有的工具、函数、流程优先复用 |
| 跳过核验 | 闭环自检 | 写完不等于做完——跑/看/查/试走一遍 |
| 破坏章法 | 遵循规范 | 风格、流程、命名——入乡随俗 |
| 假装理解 | 坦白无知 | 听不懂就说听不懂 |
| 将错就错 | 正本清源 | 看到设计问题要提，由人拍板 |

### 额外三条流程原则（各 -12.5，从总分扣）

| 原则 | 违反说明 |
|------|---------|
| 先问AI再动手 | 卡住/两次失败后没问 AI 就继续硬试 |
| 三次规则 | 同一问题尝试两次失败后，第三次没问 AI |
| 收敛原则 | 做了不必要的事情 |

### 执行步骤

1. **session_search** 查找目标时段的所有新 session（limit=20，按时间倒序）
2. **逐 session 读取完整内容**（scroll 翻页，至少看头尾 + 关键决策点）
3. **逐条对照八耻八荣**，记录每条是否被遵守；违反则注明在哪一步、什么行为
4. **评分**：每荣 +12.5，每耻 -12.5；「先错后改」可抵消（如假装理解→坦白无知 = 0）；「被指出后才改」不加分
5. **编译报告**

### 报告格式

```
## 「土同学」夜班工作评估报告
**监控时段：** YYYY-MM-DD HH:MM ~ HH:MM

### 一、总览
- 总 Session 数：N
- 累计工作时长：N 小时
- 原则综合得分：N/100

### 二、八耻八荣逐项评分
（逐条列出评估理由和分数）

### 三、各 Session 详情
（每条含时间段、任务内容、完成状态、关键违规/亮点）

### 四、主要违规汇总
（违规：时间、session、违反原则、整改建议）

### 五、综合评分与改进建议
```
