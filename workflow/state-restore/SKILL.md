# State Restore — 会话起点

## 触发

SOUL.md 要求每次新会话第一件事加载本 skill。

## 执行步骤

1. **读 SOUL.md**（已经由早课流程完成）
2. **读 CREATIVE.md** → 了解活跃上下文和开放问题
3. **读昨日日志** → `~/Documents/Obsidian Vault/每日工作记录/YYYY-MM-DD.md`
4. **收集后输出摘要**：上次做到哪、阻塞什么、今日可继续什么

## 注意

- 不要输出文件全文，只输出紧凑摘要
- 不要尝试复杂的 state 文件管理（global.yaml/projects/ 等不存在或不完整）
- 当前只有一个项目（Angelife 知识系统），不需要多项目切换
- fabric 卡片 18 张全部低质量(conf<0.5)，不自动加载
- 用户纠正时直接接受，不搞 drift detection

## 会话寿命管理

**上下文累积是输出质量的隐形杀手。** 每个来回都在往上下文里塞内容，几百条消息后 token 预算被历史吃掉，输出质量必然下降。

| 模型类型 | 安全消息数 | 超过后的表现 |
|---------|-----------|------------|
| 免费模型（如 deepseek-v4-flash-free） | ~200-300 条 | 中英混杂、句子断半截、重复绕圈 |
| 付费模型 | ~500-800 条 | 降质更慢，但最终也会出现 |

**用户明确指示：`/new` 就解决了。**

- `/new` 不是出了问题才用的补救手段，而是**预防性维护**
- 免费模型跑 200-300 条后主动 `/new`
- 付费模型跑 500+ 条后考虑 `/new`
- 不要尝试分析乱码或从中提取有用信息——直接 `/new` 重来

**关键区分：**
- 长时间会话（几百条消息）→ 上下文累积 → 输出崩 → `/new`
- 服务器常开（Hermes 网关进程）→ 不累积对话上下文 → 跑几个月没问题
- 这是两个不同的事，不要混淆
