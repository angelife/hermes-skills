# Session Lifecycle — 会话生命周期管理

## 触发条件

- 用户问"为什么输出变乱了"、"为什么中英混杂"、"为什么重复"
- 用户说 `/new` 或提到开新会话
- 会话已运行多天/数百条消息
- 讨论 Hermes 服务器 vs 会话的架构差异
- 用户问"系统一直开着会不会出问题"

---

## 核心概念

### 上下文累积定律

每个来回都在往上下文里塞内容。随着消息数增长：

| 消息量 | 效果 |
|--------|------|
| 0-100 | 正常，模型满血 |
| 100-300 | 开始吃力，输出变长/啰嗦 |
| 300-500 | 质量明显下降，中英混杂 |
| 500+ | 免费模型先崩，付费模型也降质 |

### 免费模型 vs 付费模型

| 模型类型 | 撑多久 | 表现 |
|---------|--------|------|
| 免费（如 deepseek-v4-flash-free） | 几百条后崩 | 中英混杂、句子断半截、重复绕圈 |
| 付费 | 上千条后降质 | 能撑更久，但最终也会降 |

### 会话 vs 服务器

| | 长时间会话 | 服务器常开 |
|---|---|---|
| 累积什么 | 对话历史 | 进程本身，不存对话 |
| 上下文压力 | 每轮处理全部历史 | 每个新请求独立 |
| 根因 | 上下文窗口被历史填满 | 无上下文累积 |
| 解法 | `/new` 定期重置 | 不需要处理 |

### 预防性 `/new`

- 不是出了问题才 `/new`，而是预防性使用
- 免费模型：几百条消息后考虑 `/new`
- 付费模型：上千条后考虑 `/new`
- 服务器上的 Hermes 网关模式不受此影响

## 陷阱

- 不要试图分析/修复乱码输出——直接 `/new`
- 不要混淆"长时间运行的会话"和"长时间运行的服务器进程"
- 免费模型在长上下文下输出崩是必然的，不是偶然
- 定期 `/new` 是低成本保输出质量的手段
- **cron 任务中的过度调查：** 当 cron 脚本返回空输出或文件不存在时，**3 次工具调用内必须收敛**。不要花 10+ 次调用去确认一个空文件或缺失文件。2026-07-19 04:18 教训：`wechat-group-monitor.py` 为 0 字节，助手花了 10+ 次调用才确认。正确流程：`terminal` 运行 → 空输出 → `read_file` 检查内容 → 0 字节 → 报告并输出 [SILENT]。
